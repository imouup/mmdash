# 数模Dashboard实现计划

## 目标描述
构建一个面向数学建模竞赛的协作式Dashboard平台，采用"轻量云端、重度本地"的端云协同架构。平台集成Notion作为文档与数据存储中心，Git作为实验版本追踪与隔离机制，并通过Local Agent（本地守护进程）与Cloud Agent（云端记忆中枢）实现自动化实验与AI辅助建模。用户可在Web端完成团队管理、题目解析、模型撰写与版本控制、实验设计与执行的全流程协作。

## 验收标准

遵循TDD哲学，每个验收标准均包含正面测试（应通过）与负面测试（应失败/被拒绝），以确保可确定性验证。

- AC-1: 用户与团队管理
  - 正面测试（预期通过）：
    - 新用户可通过Web界面完成注册与登录
    - 用户可创建Team并生成邀请链接，其他用户可通过链接加入
    - Team成员可在个人设置中绑定Notion账号授权
    - 团队所有者可将成员从Team中移除
  - 负面测试（预期失败/被拒绝）：
    - 未登录用户访问任何Dashboard页面应被重定向至登录页
    - 非Team成员通过伪造邀请链接加入应被拒绝
    - 普通成员（非所有者）删除Team应被拒绝
    - 使用过期或无效的Notion授权Token同步数据应报错

- AC-2: 项目创建与数据源绑定
  - 正面测试（预期通过）：
    - Team所有者可在Web端创建Project
    - 创建Project时，系统自动在Notion初始化`base_data`页面（用于主页与时间线数据）和`model_data`页面（用于模型版本备份）
    - 用户可为Project绑定Git远程仓库地址
    - 各成员可在本地Clone该仓库，并在Local Agent配置中指定仓库绝对路径
  - 负面测试（预期失败/被拒绝）：
    - 非Team成员创建该Team下的Project应被拒绝
    - 绑定无效Git仓库地址应提示错误
    - 未绑定Notion授权的用户创建Project应提示先完成授权

- AC-3: 主页面板功能
  - 正面测试（预期通过）：
    - 用户可在主页上传题目文件（PDF或纯文本），系统正确解析并展示
    - 主页展示团队Timeline概览及个人Timeline概览
    - 主页展示团队TODO与个人TODO列表
    - 主页展示整体项目进度信息
  - 负面测试（预期失败/被拒绝）：
    - 上传非PDF/文本格式文件应被拒绝或提示不支持
    - Notion `base_data`页面不可访问时，主页应显示友好的错误状态而非空白

- AC-4: 时间线面板功能
  - 正面测试（预期通过）：
    - 用户可查看类似手机日程表的团队时间安排
    - 用户可查看并设置个人时间安排
    - 时间线数据与主页`base_data`页面打通，双向同步
  - 负面测试（预期失败/被拒绝）：
    - 未授权用户查看团队时间线应被拒绝
    - Notion同步冲突时，系统应提示用户而非静默覆盖

- AC-5: 模型面板 - 基础展示与导出
  - 正面测试（预期通过）：
    - 用户可在模型面板指定Notion页面，系统定时后台拉取并展示模型文档
    - 支持将模型文档导出为PDF格式
    - 支持将模型文档导出为Markdown格式
    - 考虑到Notion API Rate Limit，服务端拉取请求应缓存至Redis/数据库，支持增量更新
  - 负面测试（预期失败/被拒绝）：
    - 指定无效的Notion页面ID应报错
    - 导出格式不在PDF/Markdown范围内应被拒绝
    - Notion API限流时，系统应使用缓存数据而非直接报错

- AC-6: 模型面板 - LLM辅助分析
  - 正面测试（预期通过）：
    - 系统通过LLM识别模型文档中的符号并生成符号表
    - 鼠标悬停在符号上方时，悬浮显示其含义（优先使用手工定义，未定义时通过上下文提供参考解释）
    - 系统通过LLM对模型进行总体和分块结构解析，并生成"模型与题目对应关系图"
    - 系统通过LLM对复杂数学公式进行拆解说明
    - 所有LLM生成的分析内容（符号表、结构解析、公式解释）强制显示"仅供参考"水印或标签
  - 负面测试（预期失败/被拒绝）：
    - LLM服务不可用时，模型面板应降级展示原始文档，而非整体崩溃
    - 未通过人工审核前，LLM分析结果不显示水印应被视为缺陷

- AC-7: 模型面板 - 版本控制与Diff
  - 正面测试（预期通过）：
    - 用户可执行Commit操作标记关键模型版本，记录修改内容
    - Diff功能可标记修改人（基于绑定的个人Notion账号识别身份）和修改内容
    - 用户可浏览历史快照并进行回滚
    - 回滚前系统自动备份当前Notion页面至`model_data`
  - 负面测试（预期失败/被拒绝）：
    - 未绑定Notion账号的用户执行Commit，无法识别身份时应标记为"未知用户"
    - 直接修改Notion原文档的功能不应存在（所有修改建议仅作为批注呈现）

- AC-8: 模型面板 - 逻辑与笔误纠正
  - 正面测试（预期通过）：
    - 系统通过LLM分析模型文档，在页面左侧以红色高亮批注标记潜在逻辑错误和笔误
    - 纠错建议以批注形式呈现，不直接修改原文档
  - 负面测试（预期失败/被拒绝）：
    - LLM误判正常内容为错误时，用户应可忽略或关闭该批注
    - 纠正批注直接修改了Notion原文档应被视为严重缺陷

- AC-9: Local Agent - 本地守护进程
  - 正面测试（预期通过）：
    - Local Agent作为守护进程在用户本机后台运行
    - 通过WebSocket（`127.0.0.1:xxxx`）与Web浏览器直接通信
    - 可检测本地环境配置（Python解释器、Conda env、gcc等）
    - 可接管本地Shell并执行命令
    - Web页面可直接向本机Local Agent发送指令，不经过云端服务器
  - 负面测试（预期失败/被拒绝）：
    - Local Agent未运行时，Web面板应提示"本地引擎未连接"而非报错
    - 非本机浏览器（跨源）直接连接Local Agent WebSocket应被拒绝或需额外授权
    - Local Agent执行危险命令（如`rm -rf /`）应被拦截或需确认

- AC-10: Cloud Agent - 云端记忆中枢
  - 正面测试（预期通过）：
    - Cloud Agent为特定Project特制，维护全局记忆
    - 实时掌握最新题目文件、模型Markdown文档、全局TODO
    - 负责下发记忆上下文给各成员的本地计算机
    - 用户上传题目文件后，Cloud Agent开始建立全局上下文向量
  - 负面测试（预期失败/被拒绝）：
    - Cloud Agent不可用时，不影响Local Agent的本地实验执行
    - Cloud Agent不应存储敏感的本地实验结果数据（实验数据留在本地Git仓库）

- AC-11: 实验与求解 - Git集成与环境
  - 正面测试（预期通过）：
    - 根据本地Git log列出各个Solver文件的修改历史
    - 用户可扫描并选择本地Git树下的任意版本、任意Solver文件
    - 系统可提取Solver文件中的可调节参数
  - 负面测试（预期失败/被拒绝）：
    - 选择非Git目录下的文件应提示错误
    - Git仓库未初始化或无可用的Solver文件时应有友好提示

- AC-12: 实验与求解 - 自动化实验执行
  - 正面测试（预期通过）：
    - 前端UI可给定参数范围和步长，计划执行序列（网格搜索）
    - Local Agent启动本地终端，依次运行参数组合
    - 实验结果自动写入本地Git目录的`/results/[时间戳]_[solver文件名]`目录
    - 目录内包含：`/fig`（图表）、`log.txt`（运行日志）、`analysis.md`（初步分析草稿）、`params_snapshot.json`（参数快照）
    - Web面板实时读取本地文件变化，展示进度和图表
  - 负面测试（预期失败/被拒绝）：
    - 参数范围或步长设置非法（如步长为0、范围为负）时应被拒绝
    - 实验执行过程中Local Agent崩溃，已生成的结果文件应保持完整
    - 同一参数组合重复执行时，新结果应覆盖或按序编号，避免数据混乱

- AC-13: 实验与求解 - 本地沙盒与团队同步
  - 正面测试（预期通过）：
    - 实验记录完全写入本地Git目录，未Push时对团队其他成员不可见
    - 用户确认实验有效后，可在本地执行`git add .`、`git commit`并`git push`
    - 其他成员Pull后，可在Web面板查看该实验记录与图表分析
    - 用户可撰写分析结论并保存至本地的`analysis.md`
  - 负面测试（预期失败/被拒绝）：
    - 未执行Git Push的实验记录被其他成员直接看到（绕过本地隔离）应视为缺陷
    - 实验结果目录结构不符合规范（缺少`fig`、`log.txt`等）时，Web面板应提示异常

## 路径边界

路径边界定义了实现质量与方案选择的可接受范围。

### 上限边界（最大可接受范围）
实现包含全部5个面板（主页、时间线、模型、实验与求解、Agent）的完整功能。Notion集成包含OAuth授权、页面初始化、内容拉取、Redis缓存与增量更新。LLM辅助分析包含符号表、结构解析、公式解释、逻辑纠错，并强制附加"仅供参考"水印。Local Agent具备完整的WebSocket通信、环境检测、Shell执行、实验自动化与结果目录生成能力。Cloud Agent具备全局记忆维护与上下文下发能力。团队协作包含完整的邀请机制、Git-based实验隔离与同步、模型版本控制与Diff。

### 下限边界（最小可接受范围）
实现核心用户注册/登录、Team/Project基础CRUD、Notion页面基础展示（主页与时间线数据读取）、模型文档的定时拉取与展示、Local Agent的基础WebSocket连接与Shell命令执行、实验的手动触发与结果记录、Git日志的基础展示。所有LLM功能可先以占位或简化形式实现（如直接调用OpenAI API返回原始文本），但架构需预留扩展空间。

### 允许选择
- **可用技术**：Next.js（前端）、FastAPI（后端）、PostgreSQL（主数据库）、Redis（缓存）、OpenAI API（LLM）、WebSocket（浏览器与Local Agent通信）、Notion API（文档存储）、GitPython或PyGit2（Git操作）、Nanobot/OpenClaw（Agent框架）
- **禁用技术/做法**：直接在Dashboard中修改Notion原文档（所有修改必须以批注形式呈现）；将本地实验数据上传至云端存储（实验数据必须保留在本地Git仓库）；使用服务端代理Local Agent指令（Web页面必须直连Local Agent）
- **确定性设计**：PRD指定了固定方案的部分（如必须基于Nanobot/OpenClaw构建Agent、所有修改建议以批注呈现、Web页面直连Local Agent），路径边界已收敛至该方案。

## 可行性提示与建议

> **注意**：本节仅供参考与理解，是概念性建议而非强制性要求。

### 概念性方案
**前端**：使用Next.js App Router构建单页Dashboard应用，各面板通过路由或标签页组织。模型面板使用Markdown渲染器展示Notion内容，配合悬浮tooltip展示符号含义。实验面板使用动态表单渲染参数输入控件。

**后端（云端）**：FastAPI提供RESTful API。PostgreSQL存储用户、团队、项目元数据。Redis缓存Notion API响应。使用Celery或APScheduler执行定时任务（如后台拉取Notion页面）。Cloud Agent基于Nanobot/OpenClaw框架构建，维护Project级上下文向量存储（可使用PostgreSQL pgvector扩展或独立向量数据库）。

**Local Agent**：Python异步守护进程（基于`asyncio`与`websockets`库），监听本地WebSocket端口（`127.0.0.1:xxxx`）。接收前端指令，执行Shell命令或Python脚本。通过`gitpython`库操作Git仓库，通过子进程调用检测本地环境（Python、Conda、gcc）。实验执行时，依次遍历参数网格，每次运行后将结果写入规范化的`/results`目录结构。

**实验流程**：前端解析Solver文件提取参数名（可通过正则或AST解析）→ 用户在UI中输入各参数的范围和步长 → 前端生成参数组合网格 → 通过WebSocket直接发送给本机Local Agent → Local Agent依次执行每个参数组合，运行Solver并将输出写入结果目录 → 前端通过文件系统监听或WebSocket推送实时读取结果文件，展示进度和图表。

**Notion缓存策略**：服务端维护一个`notion_page_cache`表（page_id, content, last_fetched_at）。每次请求先查缓存，若未命中或已过期（如超过5分钟），则调用Notion API并更新缓存。使用Redis存储序列化后的页面内容，减轻数据库压力。

### 相关参考
- `frontend/` - Next.js前端应用目录，包含各面板组件
- `backend/` - FastAPI后端服务目录，包含API路由与数据库模型
- `local_agent/` - Local Agent守护进程代码，基于Python asyncio
- `cloud_agent/` - Cloud Agent服务代码，基于Nanobot/OpenClaw框架
- 数据库模型路径：`backend/models/` - 包含users, teams, projects, team_members, notion_bindings, git_bindings等表定义

## 依赖关系与执行顺序

### 里程碑
1. **里程碑1：基础设施与用户认证**
   - 阶段A：数据库Schema设计（users, teams, projects, team_members等表）
   - 阶段B：用户注册、登录、JWT认证API
   - 阶段C：Team创建、成员邀请、Notion OAuth绑定

2. **里程碑2：Notion集成与数据同步**
   - 阶段A：Notion OAuth授权流程与Token管理
   - 阶段B：Project创建时自动初始化`base_data`和`model_data`页面
   - 阶段C：Notion内容拉取、Redis缓存、增量更新机制

3. **里程碑3：主页与时间线面板**
   - 阶段A：题目文件上传与解析（PDF/文本）
   - 阶段B：主页Dashboard（TODO、Timeline概览、进度）
   - 阶段C：时间线面板（日程表、团队与个人时间安排）

4. **里程碑4：模型面板**
   - 阶段A：Notion页面连接、定时拉取、基础展示
   - 阶段B：PDF/Markdown导出功能
   - 阶段C：LLM分析（符号表、结构解析、公式解释、纠错批注）
   - 阶段D：版本控制（Commit）、Diff、历史快照与回滚

5. **里程碑5：Agent系统架构**
   - 阶段A：Local Agent守护进程（WebSocket服务、Shell执行）
   - 阶段B：Cloud Agent（全局记忆、上下文向量、基于Nanobot/OpenClaw）
   - 阶段C：浏览器与Local Agent WebSocket直连集成

6. **里程碑6：实验与求解面板**
   - 阶段A：Git仓库绑定、Solver文件扫描、Git历史查看
   - 阶段B：本地环境检测与配置
   - 阶段C：参数提取、网格搜索UI设计
   - 阶段D：实验执行、结果目录生成、实时进度展示、`analysis.md`编辑

### 依赖说明
- 里程碑1必须在所有其他里程碑之前完成（用户认证是所有功能的前提）。
- 里程碑2必须在里程碑3和里程碑4之前完成（Notion数据是主页、时间线、模型的数据基础）。
- 里程碑5（Agent系统）与里程碑3、4可并行开发，但里程碑6（实验面板）依赖里程碑5的Local Agent。
- 里程碑4的LLM功能依赖里程碑2的Notion数据拉取。
- 里程碑6的Git功能依赖用户在本地初始化Git仓库，该操作可在里程碑1完成后由用户并行进行。

## 功能目录

---

- 主页
- 时间线
- 模型
- 实验和求解
- Agent (云端记忆中枢 & 本地执行引擎)

## 用户和项目组织方式

---

- **团队维度**: Team → Member
- **项目维度**: Team → Project

## 核心架构原则

---

- **轻量云端 (Thin Server)**: 服务器仅负责 Notion API 调用、数据状态缓存及团队基础信息存储。
- **重度本地 (Fat Client)**: 所有高强度计算、自动化实验均由部署在用户本机的 Local Agent 执行。
- **本地化实验记录**: 实验产出与 Git 仓库强绑定，未 Push 的数据对团队其他成员不可见，保障本地调试隐私与环境隔离。

# 功能面板详解

---

## 1. 主页

---

- 题目展示（PDF/纯文本解析）
- 团队 Timeline 及个人 Timeline 概览
- 团队 TODO（配合 Timeline）
- 个人 TODO
- 整体进度信息

### 数据存储

基于 Notion。创建项目时，用户需先选定一个 `base_data` 页面作为底层数据源。服务器负责轻量级同步。

## 2. 时间线

---

类似手机日程表，可设置团队和个人时间安排。

### 数据存储

基于 Notion，与主页的 `base_data` 页面打通。

## 3. 模型

---

通过连接 Notion 页面，根据最新模型文档，提供以下辅助建模功能：

- **符号表**：由 LLM 识别并标记。鼠标移至符号上方时悬浮显示含义（优先使用手工定义的含义，若未定义则通过上下文提供参考解释）。
- **结构解析**：模型总体和分块解释（基于 LLM），配合“模型 ↔ 题目”对应关系图。
- **公式解释**：基于 LLM 对复杂数学表达进行拆解说明。
- **快捷导出**：支持 PDF/Markdown 格式导出（便于喂给其他 AI 工具）。
- **版本控制 (Commit)**：便于标记模型版本和修改内容。
- **Diff 功能**：标记修改人和修改内容（需用户事先绑定个人 Notion 账号以识别身份）。
- **逻辑和笔误纠正**：位于页面左侧，用红色高亮批注标记潜在错误。
- **安全边界**：考虑到稳健性与防冲突，暂不提供直接在 Dashboard 修改 Notion 原文档的功能，所有修改建议作为“批注”呈现。

> **注：模型和公式解释在人工审核前，强制打上“仅供参考”水印/标签。**
> 

### 数据存储

直接存在 Notion。创建项目时用户需选定一个 `model_data` 页面作为各版本模型文档的 Backup 位置。
**性能优化**：考虑到 Notion API 的 Rate Limit，所有的拉取请求发生在服务端，并存入 Redis/数据库做适当缓存与增量更新。

## 4. 实验和求解 (本地化工作流)

---

### 功能

- **环境接管**：通过 Local Agent 检测本地环境配置，设定工作环境（指定 Python 解释器、Conda env、gcc 等）。
- **版本追踪**：根据本地 Git log 列出各个 Solver 文件的修改历史。
- **自动化实验设计与执行**：
    - 选取当前本地 Git 树下的任意版本、任意 Solver 文件。
    - 提取可调节参数，通过前端 UI 给定参数范围和步长，计划执行序列。
- **本地沙盒化记录**：
    - 记录实验过程、结果以及结果文件位置。
    - **隔离性说明**：实验记录完全写入本地 Git 目录。若当前用户未执行 `git push`，其他团队成员的 Dashboard 无法看到该最新记录。

> **规范化实验结果目录结构：**
目录名：`/results/[时间戳]_[solver文件名]`
目录内容：`/fig` (图表), `log.txt` (运行日志), `analysis.md` (初步分析草稿), `params_snapshot.json` (参数快照)
> 

### 数据存储

本地 Git 仓库。创建项目时用户需在 Web 端绑定远程 Git 仓库地址，并在本地指定 Clone 路径。

## 5. Agent (端云协同)

---

采用“云端大脑 + 本地执行手脚”的拆分架构，基于 Nanobot / OpenClaw 构建。

- **Cloud Agent (云端记忆中枢)**：
    - 为特定 Project 特制。
    - **全局记忆维护**：实时掌握最新的题目文件、模型 Markdown 文档、全� TODO。
    - 负责下发记忆上下文给各成员的本地计算机。
- **Local Agent (本机守护进程)**：
    - 在用户本地后台运行，通过 WebSocket (`127.0.0.1:xxxx`) 与 Web 浏览器直接通信。
    - **能力**：接管本地 Shell、执行自动化实验、将云端下发的“模型解释”与本地跑出的“实验结果”结合，自动生成本地的实验分析报告。

# 用户使用流程示例

---

### 开始使用

1. 注册 Web 账户。
2. 登录账户并创建 Team，发送邀请链接给队员。
3. 团队成员各自绑定 Notion 授权。
4. **安装本地引擎**：所有成员在本地电脑运行/安装 Local Agent 客户端，并使其在后台运行。

### 赛前准备

1. 队长在 Web 端创建 Project。
2. 绑定 Notion 页面（竞赛协作文档根页面），系统自动初始化 `base_data` 和 `model_data` 页面。
3. 绑定 Git 远程存储库。各成员在自己的电脑上 Clone 该仓库，并告诉本地的 Local Agent 该仓库的绝对路径。

### 竞赛开始 (实战演练)

1. **题目解析**：用户于首页上传题目文件（PDF 和 Data），Cloud Agent 开始建立全局上下文向量。
2. **任务分配**：用户设定 Timeline，撰写个人与团队 TODO。
3. **模型推导**：用户开始在 Notion 书写模型，并在“模型”面板指定该页面。“模型”面板开始定时在后台拉取、缓存并备份模型。
4. **模型定版**：用户 Commit 关键模型版本。其他成员在“模型”面板点击“分析模型”，生成结构图、符号表和公式解释。
5. **本地代码实验 (核心环节)**：
    - 负责编程的队员 A 打开 Web 面板，页面自动通过 WebSocket 连通 A 本机的 Local Agent。
    - 队员 A 点击“扫描”获取本地 Solver 文件，选择目标文件（如 `solver_v2.py`）及对应的 `params.py`。
    - 页面解析出参数，队员 A 设定网格搜索的范围和步长。
    - 队员 A 点击“开始实验”。
    - **指令下发**：Web 页面直接向本机的 Local Agent 发送执行指令（不经过云端服务器）。
    - **本地执行**：Local Agent 启动本地终端，依次运行参数，自动在本地 Git 的 `/results` 目录下生成规范化结果（日志、图片）。
    - Web 面板实时读取本地文件变化，展示进度和图表。
    - 队员 A 撰写分析结论，保存至本地的 `analysis.md`。
6. **团队成果同步**：
    - 队员 A 确认实验有效，在本地执行 `git add .`、`git commit` 并 `git push`。
    - 队员 B 的 Local Agent 监听到远端更新（或 B 手动 Pull）后，B 的 Web 面板才会显示出 A 刚才的实验记录与图表分析。
7. **模型回滚与迭代**：发现思路有误，用户在“模型”面板浏览历史快照，备份当前 Notion 页面，并将旧版复制恢复，循环迭代。�
